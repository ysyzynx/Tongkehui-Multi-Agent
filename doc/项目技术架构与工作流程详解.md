## 项目概述

### 什么是"童科绘"？

**童科绘**是一个**多智能体协作的Web端AI辅助科普内容创作平台**。它模拟了一个完整的内容创作团队，包括：

- 📝 **故事原创者** - 创作科普故事
- 🔬 **科学审核者** - 审核科学准确性
- ✒️ **文学润色者** - 提升文学性
- 👀 **虚拟读者** - 模拟目标受众反馈
- 🎨 **插画师** - 生成绘画提示词和图片
- 🖨️ **排版师** - 生成HTML和PDF

通过这个多智能体协作系统，用户只需输入主题、受众等基本参数，就能获得一篇高质量、图文并茂的科普绘本。

```python
app = FastAPI(title="童科绘", version="1.0.0")

#### pipeline_runner.py - 命令行全自动工作流

**功能**：
- 交互式获取用户参数（主题、受众、风格等）
- 按顺序调用6个智能体
- 展示每一步的中间结果

### 2. 智能体层（agent/）

#### base_agent.py - 智能体抽象基类

**设计模式**：模板方法模式 + 抽象基类

**功能**：
- 定义所有智能体的统一接口
- 提供通用的上下文管理
- 规范 `run()` 方法签名

```python
class BaseAgent(ABC):
    def __init__(self, name: str, description: str):
        self.name = name
        self.description = description
        self.context = {}  # 上下文存储
        self.result = None

    @abstractmethod
    def run(self, *args, **kwargs) -> Any:
        """子类必须实现的核心方法"""
        pass
```

#### story_creator.py - 故事创作智能体

**职责**：根据用户参数生成完整的科普故事

**输入参数**：
- `theme`: 科普主题（如"黑洞"、"光合作用"）
- `age_group`: 年龄段（如"6-12岁"）
- `style`: 文章风格（如"趣味科普"、"冒险故事"）
- `target_audience`: 目标受众群体
- `extra_requirements`: 额外要求（如"主角是小猫"）
- `word_count`: 目标字数

**输出**：
```json
{
  "title": "小水滴的天空旅行",
  "content": "第一章 神秘的消失\n清晨的阳光..."
}
```

**提示词优化要点**：
- 增加"优秀科普故事五大特征"
- 提供章节结构建议
- 增加思维链引导（先思考后输出）
- 提供JSON输出示例

#### science_checker.py - 科学审核智能体

**职责**：审核科学准确性，提取专业术语

**输入参数**：
- `story_title`: 文章标题
- `story_content`: 文章正文
- `target_audience`: 目标受众
- `evidence_context`: 检索到的科学证据（可选）
- `deepsearch_context`: DeepSearch分析结果（可选）

**输出**（12个字段）：
```json
{
  "passed": true,
  "issues": ["问题1", "问题2"],
  "modifications_made": ["修改1", "修改2"],
  "suggestions": "结构化评审结论...",
  "review_sections": [
    {
      "section": "事实准确性校验",
      "status": "通过",
      "finding": "...",
      "suggestion": "...",
      "suggested_revision": "...",
      "adopted": false
    }
  ],
  "highlight_terms": ["术语1", "术语2"],
  "revised_content": "修正后的正文...",
  "glossary": [
    {"term": "核聚变", "explanation": "..."}
  ],
  "revised_glossary": [...],
  "evidence_used": [...]
}
```

**审核四大维度**：
1. **事实准确性校验** - 检查科学概念、数值是否正确
2. **专业术语适用性检查** - 评估术语难度是否匹配受众
3. **科学逻辑验证** - 检查因果关系、推理过程
4. **引用来源建议** - 建议权威引用来源

**后处理逻辑**：
- 标准化术语提取（过滤非科学词汇）
- 确保高亮词8-12个，词汇表4-6个
- 结构化评审章节

---

#### literature_checker.py - 文学润色智能体

**职责**：从文学角度审查并润色故事

**输入参数**：
- `title`: 故事标题
- `content`: 故事正文

**输出**：
```json
{
  "passed": true,
  "feedback": "综合评价...",
  "revised_content": "润色后的完整故事..."
}
```

**四大评审角度**：
1. **语言表达** - 科学性与文采的平衡
2. **叙事结构** - 起因、经过、高潮、结果是否完整
3. **节奏张力** - 情节起伏、知识植入是否自然
4. **引人入胜的开头** - 是否设置悬念

---

#### reader.py - 虚拟读者智能体

**职责**：模拟目标受众的阅读反馈

**输入参数**：
- `story_content`: 故事正文
- `title`: 故事标题
- `target_audience`: 目标受众身份

**输出**：
```json
{
  "reader_feedback": "我的身份是...\n1. 易读性：...\n2. 科普性：...\n..."
}
```

**五大反馈维度**：
1. **易读性** - 能否轻松看懂
2. **科普性** - 有没有学到新知识
3. **有趣性** - 故事好不好玩
4. **实用性** - 知识对生活有没有帮助
5. **传播性** - 想不想分享给别人

**特色设计**：
- **角色沉浸热身** - 先让LLM"代入"角色再评价
- **第一人称表达** - 必须使用"我"
- **降级策略** - JSON失败时改用纯文本生成

#### illustrator.py - 插画师智能体

**职责**：切分场景，生成绘画提示词和图片

**输入参数**：
- `story_content`: 故事正文
- `image_count`: 分镜数量
- `art_style`: 艺术风格
- `extra_requirements`: 额外绘画要求

**输出**：
```json
[
  {
    "scene_id": 1,

    "summary": "画面大意...",
    "image_prompt": "详细的绘画提示词...",

**画面元素清单**（提示词优化新增）：
1. **主体** - 角色外貌、动作、穿着
2. **环境** - 场景、背景、季节天气
3. **光影** - 光源方向、光线类型、阴影
4. **色彩** - 主色调、氛围色、饱和度
5. **构图** - 镜头角度、景别、画面比例
6. **艺术风格** - 画风描述

**关键逻辑**：
- `_balanced_semantic_chunks()` - 智能语义分块
- 调用火山引擎/通义万相生图
- 支持单张图片重绘

    db = SessionLocal()

#### pdf_generator.py - PDF生成工具

**双方案支持**：
1. **优先方案**：WeasyPrint（快速、轻量）
2. **降级方案**：Playwright（兼容性好）

```python
@staticmethod
def html_to_pdf(html_content, output_path):
    try:
        # 方案1：WeasyPrint
        from weasyprint import HTML
        HTML(string=html_content).write_pdf(output_path)
    except:
        # 方案2：Playwright
        with sync_playwright() as p:
            browser = p.chromium.launch()
            page.pdf(path=output_path)
```

## 技术原理深度解析

### 1. 提示词工程（Prompt Engineering）

#### 为什么提示词如此重要？

LLM的输出质量 = f(模型能力 × 提示词质量)

即使使用相同的模型，好的提示词可以让输出质量提升30-50%。

#### 本次优化的核心策略

| 策略 | 原理 | 效果 |
|------|------|------|
| **思维链（CoT）** | 让LLM先思考再输出，模拟人类推理过程 | 减少错误，提升逻辑 |
| **示例驱动** | 提供具体的"好vs坏"例子 | LLM更容易理解要求 |
| **结构化检查清单** | 明确的验证步骤 | 减少遗漏，提升完整性 |
| **角色沉浸** | 先让LLM"代入"角色 | 反馈更真实 |
| **输出格式示例** | JSON示例 | 降低解析失败率 |

#### 提示词模块化的好处

```
修改前：
┌─────────────────────────────────┐
│ agent/story_creator.py          │
│  - 200行代码                    │
│  - 100行是硬编码的提示词        │
│  - 修改提示词需要改代码          │
└─────────────────────────────────┘

修改后：
┌─────────────────────────────────┐
│ prompts/story_creator_prompt.py │ ← 独立管理
│  - SYSTEM_PROMPT               │
│  - USER_PROMPT_TEMPLATE        │
└─────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────┐
│ agent/story_creator.py          │ ← 只包含逻辑
│  - 导入提示词模板               │
│  - 组装参数                     │
│  - 调用LLM                      │
└─────────────────────────────────┘
```

好处：
- 提示词和代码分离，职责清晰
- 可以独立版本控制提示词
- 便于A/B测试不同提示词
- 非技术人员也可以调整提示词

---

### 2. 多智能体协作原理

#### 为什么用多个智能体而不是一个？

**单智能体方案的问题**：
```
用户 → LLM → 输出
     (一个模型做所有事情)
```
- 任务太复杂，LLM难以兼顾所有方面
- 没有审核机制，错误容易累积
- 难以针对特定环节优化

**多智能体方案的优势**：
```
        ┌─────────────┐
        │ StoryCreator │ → 专注创作
        └─────────────┘
              │
              ▼
        ┌──────────────────┐
        │ LiteratureChecker│ → 专注文学性
        └──────────────────┘
              │
              ▼
        ┌─────────────────┐
        │ ScienceChecker  │ → 专注科学性
        └─────────────────┘
              │
              ▼
        ┌──────────┐
        │  Reader  │ → 专注用户体验
        └──────────┘
              │
              ▼
        ┌─────────────┐
        │ Illustrator │ → 专注视觉
        └─────────────┘
```
- 每个智能体"专家化"，提示词可以针对特定任务优化
- 串行流水线，前一步的输出是后一步的输入
- 每个环节都有反馈，可以回滚和重新生成
- 模拟真实的内容创作团队

### 3. LLM调用封装原理

#### llm_client.py 的设计

```python
class LLMClient:
    def __init__(self):
        # 使用OpenAI兼容接口
        self.client = OpenAI(
            api_key=settings.OPENAI_API_KEY,
            base_url=settings.OPENAI_API_BASE
        )

    def generate_json(self, system_prompt, user_prompt):
        try:
            response = self.client.chat.completions.create(
                model=self.model,
                messages=[
                    {"role": "system", "content": system_prompt},
                    {"role": "user", "content": user_prompt}
                ],
                temperature=0.7,  # 创造性参数
            )

            content = response.choices[0].message.content

            # 自动清洗Markdown标记
            if content.startswith("```json"):
                content = content[7:]
            if content.endswith("```"):
                content = content[:-3]

            return json.loads(content)

        except Exception as e:
            return {"error": str(e)}
```

**关键点**：
1. **OpenAI兼容接口** - 可以无缝切换通义千问、文心一言等
2. **temperature=0.7** - 平衡创造性和稳定性
3. **自动清洗** - 处理LLM习惯性添加的Markdown标记
4. **错误兜底** - 异常时返回结构化错误对象

### 5. PDF生成原理

#### WeasyPrint vs Playwright

| 特性 | WeasyPrint | Playwright |
|------|-----------|-----------|
| 速度 | 快（直接渲染） | 较慢（启动浏览器） |
| 依赖 | 需要GTK库 | 需要Chromium |
| CSS支持 | 部分支持 | 完整支持 |
| 资源占用 | 轻量 | 较重 |

**双方案设计**：
```python
try:
    # 优先用WeasyPrint
    HTML(string=html).write_pdf(output_path)
except:
    # 失败时降级到Playwright
    with sync_playwright() as p:
        browser = p.chromium.launch()
        page.pdf(path=output_path)
```
